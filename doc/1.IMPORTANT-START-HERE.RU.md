# Отчёт о внедрении: корпоративный сайт WellnessLiving (Astro + GitHub Pages)

> **⚠️ ВАЖНО — Прочитайте перед принятием решения о внедрении**
>
> Обещание «менеджеры редактируют контент без разработчика» работает **только после того, как разработчик сначала построит ядро системы**. Astro — это фреймворк на **Node.js + TypeScript**: вёрстка, стили, маршрутизация, конфигурация сборки и пайплайн GitHub Actions — всё это работа разработчика. Простота, которую видят менеджеры, — это **результат** этой предварительной инженерной работы, а не бесплатная возможность платформы.
>
> **Что это значит для бюджета:**
> - **Первичная настройка — нужен разработчик.** Создание и конфигурация проекта Astro, макетов, стилей, логики навигации и CI/CD-пайплайна. Разовая работа — от нескольких дней до пары недель в зависимости от объёма дизайна.
> - **Периодическое сопровождение — нужен разработчик.** Обновления фреймворка, изменения дизайна, новые функции (формы, поиск, интеграции, аналитика), починка сломанной сборки, любые структурные или макетные изменения.
> - **Ежедневная работа с контентом — ведут менеджеры.** Когда ядро стабильно, менеджеры самостоятельно добавляют страницы, посты, изображения и пункты меню. **В этом и ценность подхода** — но она раскрывается только после того, как инженерная работа выполнена.
>
> **Про помощь AI (Claude Code / OpenCode / Copilot):** AI-инструменты *могут* выполнять работу по Astro / Node.js / TypeScript — писать компоненты, чинить сборку, мигрировать код, обновлять зависимости. **Но только при наличии разработчика, способного вести предметный и технически корректный диалог** по фреймворку, по экосистеме JavaScript/TypeScript и по архитектуре проекта. AI **усиливает** разработчика — он его не заменяет. Менеджер или нетехнический оператор не сможет корректно направлять AI в решениях по Astro/Node/TS — результатом будут сломанные сборки, незаметные регрессии и накапливающийся технический долг. Строка бюджета — это не «AI-инструмент», а «разработчик, умеющий пользоваться AI в этом стеке».
>
> Если вы рассчитываете на **нулевое** участие разработчика на любом этапе, это решение не подойдёт.

## 1. Обзор решения

Решение представляет собой статический корпоративный сайт на фреймворке Astro, размещённый на GitHub Pages и автоматически разворачиваемый при каждом изменении контента. Контент-менеджеры редактируют Markdown-файлы и JSON-конфигурацию прямо в репозитории GitHub (через браузер или VS Code), фиксируют изменения — сайт пересобирается и публикуется автоматически за 30–60 секунд без участия разработчиков. Сайт поддерживает страницы, блог, вложенную навигацию с выпадающими меню и полностью адаптивен.

### Схема процесса

```
Менеджер редактирует .md / .json файл
         │
         ▼
   git commit & push
   (GitHub UI или VS Code)
         │
         ▼
  GitHub Actions запускается
  (автоматическая сборка ~30 сек)
         │
         ▼
  Статический сайт развёрнут
  на GitHub Pages
         │
         ▼
  Сайт обновлён
  (без сервера, без CMS, без базы данных)
```

## 2. Ключевые особенности внедрения

### 2.1 GitHub Pages и видимость репозитория

GitHub Pages на **бесплатных организационных планах** работает только с **публичными репозиториями**. Репозиторий `wellnessliving/astro-1st` не содержит исходного кода приложений, секретов или конфиденциальных данных — только контент сайта (тексты, изображения, стили). Два варианта:

| Вариант | Стоимость | Компромисс |
|---------|-----------|-----------|
| **(а) Сделать репозиторий публичным** | Бесплатно | См. предупреждение о безопасности ниже. |
| **(б) Обновить план до GitHub Team** | $4/пользователь/месяц | Репозиторий остаётся приватным. GitHub Pages работает с приватными репо на платных планах. |

При внедрении был выбран вариант (а) — репозиторий сделан публичным.

> **⚠️ ПРЕДУПРЕЖДЕНИЕ О БЕЗОПАСНОСТИ — ПУБЛИЧНЫЙ РЕПОЗИТОРИЙ**
>
> Публичный репозиторий означает, что **весь исходный код, все файлы контента, история коммитов и каждое когда-либо сделанное изменение навсегда видны любому человеку в интернете.** Это включает:
>
> - **Все Markdown-файлы** — каждая страница, пост блога, черновик и удалённый контент (восстановим из истории git)
> - **Структура навигации** (`navigation.json`) — полная карта сайта
> - **CSS и код вёрстки** — полный исходный код дизайна/темы
> - **Конфигурация GitHub Actions** — пайплайн деплоя
> - **Сообщения коммитов и информация об авторах** — кто что менял и когда (имена, email-адреса)
> - **Удалённый или откаченный контент** — всё, что когда-либо было закоммичено, навсегда остаётся в истории git, даже после удаления
> - **AI-контекстные файлы** (`CLAUDE.md`, `AGENTS.md`) — архитектура и соглашения проекта
>
> **Что это значит на практике:**
> - Конкуренты могут видеть структуру сайта, контент-стратегию и готовящиеся страницы до их публикации
> - Поисковые системы могут индексировать сырое содержимое репозитория наряду с сайтом
> - Любые случайно закоммиченные секреты (API-ключи, пароли, внутренние URL) становятся мгновенно и навсегда публичными
>
> **НИКОГДА не размещайте в репозитории:**
> - API-ключи, токены, пароли или учётные данные любого рода
> - Внутренние документы компании, финансовые данные или информацию о сотрудниках
> - Черновой контент, являющийся конфиденциальным или находящийся под эмбарго
> - Данные клиентов, ключи аналитики или учётные данные сторонних сервисов
>
> **Если конфиденциальность важна — перейдите на GitHub Team ($4/пользователь/месяц) для приватного репозитория.** Это рекомендуемый подход для любой организации, которая считает контент, структуру или историю редактирования своего сайта конфиденциальной информацией.

### 2.2 Домены и поддомены

| Конфигурация | DNS-запись | Примечания |
|-------------|-----------|-----------|
| Поддомен (напр. `www.example.com`) | CNAME → `wellnessliving.github.io` | Рекомендуется. Простая настройка. |
| Корневой домен (напр. `example.com`) | A-записи → IP-адреса GitHub (4 адреса) | Работает, но без CDN-преимуществ у некоторых DNS-провайдеров. |
| Подпуть (текущий: `wellnessliving.github.io/astro-1st/`) | DNS не требуется | Бесплатный вариант по умолчанию. Подходит для тестирования. |

Ключевые моменты:
- Каждый GitHub Pages сайт организации занимает один поддомен или корень `orgname.github.io`.
- HTTPS обеспечивается автоматически GitHub для пользовательских доменов.
- При переходе на свой домен необходимо обновить (убрать) параметр `base` в `astro.config.mjs` и скорректировать внутренние ссылки. Это разовая задача разработчика (~15 минут).

> **⚠️ ВАЖНО: Один репозиторий = один домен/поддомен**
>
> GitHub Pages имеет фундаментальное архитектурное ограничение: **каждый репозиторий может обслуживать только один домен (или поддомен)**. Невозможно разместить несколько поддоменов из одного репозитория.
>
> Если организации нужны несколько поддоменов (напр. `blog.example.com`, `docs.example.com`, `support.example.com`), **каждый поддомен требует собственного отдельного репозитория** со своим проектом Astro, своим `navigation.json`, своим GitHub Actions workflow и своими файлами контента. Они не делят вёрстку, стили или навигацию.
>
> **Что это значит для планирования:**
> - Один корпоративный сайт на `www.example.com` — один репозиторий (текущая конфигурация) ✅
> - Добавить отдельный `blog.example.com` — второй репозиторий, второй проект, отдельное обслуживание ⚠️
> - 5 поддоменов = 5 репозиториев = 5× затраты на обслуживание ⚠️
>
> **Альтернативы, если нужно несколько поддоменов:**
> - Использовать подпути вместо поддоменов (`example.com/blog/`, `example.com/docs/`) — работает из одного репо ✅
> - Использовать другую хостинг-платформу (Netlify, Cloudflare Pages, Vercel), которая может обслуживать несколько путей из одного проекта
> - Использовать монорепо с раздельными результатами сборки (продвинутый вариант, требует кастомного CI/CD)

### 2.3 Ограничения для менеджеров и сложность эксплуатации

| Область | Что менеджер МОЖЕТ | Что менеджер НЕ МОЖЕТ |
|---------|--------------------|-----------------------|
| **Контент** | Редактировать текст, добавлять страницы, посты, картинки, создавать разделы с выпадающими меню | Менять макет страницы, добавлять интерактивные элементы (формы, виджеты), изменять дизайн |
| **Навигация** | Добавлять/удалять/перемещать пункты меню, создавать подменю | Менять структуру шапки/подвала, изменять поведение CTA-кнопки |
| **Деплой** | Контент публикуется автоматически при коммите | Нет ручного контроля деплоя, нет staging-окружения, нет предпросмотра перед публикацией |

Факторы операционной сложности:
- **Необходимо знание Markdown.** Менеджеры должны освоить базовый синтаксис Markdown (жирный, заголовки, ссылки, картинки, таблицы). Шпаргалка включена в документацию.
- **Необходим аккаунт GitHub.** Каждому контент-менеджеру нужен GitHub-аккаунт с правами записи в репозиторий.
- **Нет визуального редактора.** Нет WYSIWYG-редактора — контент редактируется как обычный текст. VS Code с предпросмотром Markdown частично решает проблему, но это не drag-and-drop интерфейс.
- **Редактирование JSON подвержено ошибкам.** Изменение навигации требует редактирования JSON-файла. Пропущенная запятая или скобка ломает сборку сайта. Менеджеры должны точно следовать документированному формату.
- **Нет предпросмотра перед публикацией.** Изменения публикуются немедленно после коммита. Режима «черновик» или «staging» нет, если только менеджер не запускает сайт локально через `npm run dev` (продвинутый workflow, требует Node.js).
- **Нет интерфейса отката.** Если менеджер что-то сломал, восстановление требует либо отката коммита через GitHub (умеренно техническая операция), либо помощи разработчика.

## 3. Риски

| # | Риск | Влияние | Снижение |
|---|------|---------|---------|
| 1 | **Менеджер ломает сайт** невалидным JSON или Markdown | Сборка падает, текущая версия остаётся live, новые изменения не появляются | **AI-режим (Claude Code / OpenCode):** AI валидирует JSON и запускает `npm run build` перед каждым коммитом — сломанный код не попадёт в репо. Ручной режим: документация с шаблонами предоставлена; разработчик может откатить коммит за <5 мин |
| 2 | **Нет staging/предпросмотра** — ошибки публикуются сразу | Посетители могут кратковременно видеть сломанный контент | Локальный предпросмотр для продвинутых менеджеров (VS Code + `npm run dev`); ошибка сборки предотвращает деплой сломанного кода |
| 3 | **Привязка к GitHub** | Усилия на миграцию при изменении ценовой политики GitHub | Astro генерирует стандартный HTML; сайт можно перенести на любой статический хостинг (Netlify, Cloudflare Pages, S3) с минимальными усилиями |
| 4 | **Публичный репозиторий показывает историю изменений** | Удалённый или отредактированный контент остаётся видимым в истории git | Приемлемо для маркетингового контента; конфиденциальная информация не должна размещаться в репозитории |
| 5 | **Адаптация менеджеров** — Markdown + GitHub может быть слишком техническим | Медленное обновление контента, продолжение зависимости от разработчиков | Обучение (1–2 часа); полная документация предоставлена на EN и RU; workflow через VS Code с визуальным предпросмотром как промежуточный вариант |
| 6 | **Единая точка отказа: GitHub** | Сайт недоступен при сбое GitHub Pages | Историческая доступность GitHub 99.9%+; для критически важных сайтов — CDN-зеркало (Cloudflare) |
